# GEMINI.md

이 문서는 Gemini(Google AI)가 본 프로젝트에서  
**GitHub Issue, Pull Request, Release Note**를 작성할 때  
반드시 따라야 할 협업 규칙과 문서 형식을 정의합니다.

---

## 🤝 협업 컨벤션 (Convention)

### 브랜치 전략 (Git Flow)

아래 브랜치 전략을 기준으로 작업 내용을 작성합니다.

- **main**
  - 배포 가능한 안정 버전 (Production)
- **dev**
  - 다음 배포를 위한 통합 개발 브랜치
- **feat/기능명(이슈명)**
  - 새로운 기능 개발
  - 예: `feat/base-entity`
- **fix/기능명(이슈명)**
  - 버그 수정
  - 예: `fix/inventory-error`

---

## 📝 커밋 메시지 규칙

커밋 메시지는 반드시 아래 타입 중 하나로 시작합니다.

- **feat**: 새로운 기능 추가
- **fix**: 버그 수정
- **docs**: 문서 수정 (README, Swagger 등)
- **test**: 테스트 코드 추가
- **refactor**: 코드 리팩토링 (기능 변경 없음)
- **chore**: 설정, 빌드, 패키지 매니저 변경

### 예시

- `feat: 공통 BaseEntity 계층 구현`
- `refactor: 엔티티 감사 필드 구조 정리`

## 🐞 GitHub Issue 작성 가이드

Gemini는 GitHub Issue 작성 시 아래 형식을 **반드시 유지**합니다.

### 필수 섹션

1. **✨ 어떤 기능인가요?**
   - 한 줄 요약 형태
   - 무엇을 만드는지 명확히 설명

2. **🔨 작업 상세 내용**
   - 체크리스트(`- [ ]`) 형식 사용
   - 구현 단위별로 세분화

3. **💡 작업 목적**
   - 왜 이 작업이 필요한지 설명
   - 기술적 이유 또는 협업 관점의 목적 포함

### 작성 규칙

- 감정 표현, 구어체 사용 금지
- 결과를 단정하지 않고 “계획/의도” 중심으로 작성
- PR에서 체크리스트 완료 여부 갱신 가능

---

## 📌 Pull Request 작성 가이드

PR에는 다음 항목을 포함합니다.

- **관련 이슈**
- **작업 내용 요약**
- **테스트 결과**
- **리뷰 요청 사항(선택)**

PR 설명은 “무엇을 / 왜 / 어떻게” 변경했는지가 드러나야 합니다.

---

## 🗒️ 릴리스 노트 작성 가이드

릴리스 노트는 변경 사항의 **의도와 효과**가 드러나도록 작성합니다.

- 기술적인 변경 사항을 숨기지 않습니다.
- 사용자 또는 시스템 관점의 개선점을 함께 서술합니다.

### 예시

**Chores**

- 공통 엔티티 계층 개선으로 감사 추적 및 소프트 삭제 기능 지원
- 데이터 관리 안정성 및 유지보수성 향상
